# Phase 16 - UI Design Contract

> Visual and interaction contract for the Component Library phase.
> Generated by gsd-ui-researcher for `gsd-planner`, `gsd-executor`, `gsd-ui-checker`, and `gsd-ui-auditor`.
> Sources: `.planning/STATE.md`, `.planning/ROADMAP.md`, `.planning/REQUIREMENTS.md`, `16-CONTEXT.md`, `16-RESEARCH.md`, `15-UI-SPEC.md`, `app/globals.css`, `components/ui/card.tsx`, `components/features/raids/raid-card.tsx`, `components/features/hero/hero-card.tsx`, `components/ui/hex-radar.tsx`, `components/features/captain/d20-roll-animation.tsx`, `components/features/raids/roll-hero-reveal.tsx`, `app/onboarding/page.tsx`.

---

## Scope

| Area | Contract |
|------|----------|
| In scope | New UI primitives, additive upgrades to existing cards/charts/reveal components, onboarding step progress, visual polish that is immediately visible on live raid cards and onboarding |
| Out of scope | Full page redesigns, auth layout work, background isolation, particles, sound, GSAP, typewriter, ScrollTrigger, new product capability |
| Inheritance rule | Phase 16 must inherit Phase 15 Emerald Guild system rules instead of redefining the token layer |
| Design priority | Make the new direction obvious first on `RaidCard`, then onboarding progress, then hero and Roll Hero surfaces |

Phase 16 is a component phase, not a new design-system phase. The executor must reuse the Emerald Guild spacing, typography, color, glass, and motion rules already locked in Phase 15 and only define how new and upgraded components consume them.

## Design System

| Property | Value |
|----------|-------|
| Tool | none - repo-native CSS tokens in `app/globals.css` |

`components.json` is absent and this phase explicitly stays inside the repo-native design system. Registry safety is therefore not applicable beyond recording that no registry-backed blocks are allowed in Phase 16.

## Inheritance From Phase 15

These rules are locked and must not be redefined in this phase:

- Spacing scale, typography scale, base color roles, glow restraint, reduced-motion behavior, and z-index ladder inherit directly from `15-UI-SPEC.md`.
- `RaidCard` is the only Phase 16 surface allowed to use the cursor-following spotlight effect.
- Hero, captain, onboarding, and shell surfaces use calmer elevation than raid discovery cards.
- Cinzel stays limited to titles, headings, and brand moments; badges, progress labels, percentages, metadata, and helper copy stay on sans/mono contracts.
- Existing `.dr-card`, `.dr-card--elevated`, `.dr-card--glow-emerald`, `.dr-heading`, `.dr-label`, `.dr-btn-primary`, and `.dr-btn-secondary` contracts remain the canonical styling layer.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Rune accents, inline separators, micro icon padding |
| sm | 8px | Chip gaps, slot gaps, compact metadata spacing |
| md | 16px | Default component padding, internal card stacks, badge groups |
| lg | 24px | Card section spacing, onboarding panel breathing room |
| xl | 32px | Gaps between major card groups and adjacent modules |
| 2xl | 48px | Large feature separation inside hero and onboarding surfaces |
| 3xl | 64px | Page-level spacing for Phase 17 consumers of these components |

Exceptions: hex-slot glyphs may render at 14-16px visual size inside dense slot clusters; touchable icon/button targets still stay at or above 44px.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 12px | 600 | 1.4 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px base | 600 | 1.1 |

Phase 16 typography rules:

- Inherit the Phase 15 font stack unchanged.
- Use Cinzel only for card titles, section titles, and Roll Hero result headings.
- Use JetBrains Mono for fit scores, progress labels, slot labels, and compact metadata.
- Use Sora for body copy, helper copy, and interactive labels.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#05070D` | Deep canvas behind all cards and onboarding containers |
| Secondary (30%) | `#22302C` | Card bodies, elevated panels, calmer hero/captain surfaces |
| Accent (10%) | `#00D97E` | Raid card hover ring, spotlight center, active step node, skill-bar fill, radar profile stroke, avatar glow ring, primary CTA fill |
| Destructive | `#EF4444` | Destructive actions only |

Accent reserved for: `RaidCard` hover emphasis, spotlight overlay core, active onboarding step state, `SkillBar` fill, primary `HexRadar` polygon stroke/fill, `AvatarGlow` halo, selected controls, and primary CTA emphasis only.

Support color rule: gold `#C8943A` is the secondary highlight for premium fit scores, completed steps, premium dividers, and high-value result accents. It is not a substitute for the emerald primary accent.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Join Raid |
| Empty state heading | No raids match this path |
| Empty state body | Adjust your filters or check back after new raids are posted. |
| Error state | We couldn't load this board. Refresh the page or try again in a moment. |
| Destructive confirmation | none in Phase 16 |

Copy tone rules:

- Keep actionable UI copy plain and scan-friendly.
- Use fantasy flavor in titles and section names, not in recovery/error language.
- Onboarding action labels stay direct: `Next` and `Finish`.
- Roll Hero action copy should mature to `Invite Hero` and `Re-roll`, not introduce lore-heavy verbs.

## New Component Contracts

### `FitScoreBadge`

| Property | Contract |
|----------|----------|
| Tier mapping | Gold at `>= 80`, emerald at `60-79`, muted below `60` |
| Shape | Compact pill or soft-hex pill; never a large card-like badge |
| Typography | JetBrains Mono label, uppercase or compact sentence case, 12px max |
| Visual emphasis | Gold is the highest tier; do not invert the hierarchy by making emerald the best tier |
| Placement | Right-side meta cluster on cards and result surfaces |

### `HexSlotIndicator`

| Property | Contract |
|----------|----------|
| Geometry | Inline SVG hexagon; pointed, clean, and readable at small sizes |
| Filled state | Role-colored stroke with subtle tinted fill |
| Empty state | Same geometry with transparent fill and reduced opacity |
| Density | Default 16px visual size with 4px gap between slot glyphs |
| Label pairing | Each slot cluster keeps a short mono label below or adjacent; do not rely on color alone |

### `SkillBar`

| Property | Contract |
|----------|----------|
| Track | 8px height on default cards, rounded ends, muted secondary track |
| Fill | Emerald primary fill with optional gold marker only for comparative or target overlays |
| Motion | Animate from 0 to target width once on first viewport entry |
| Reduced motion | Render directly at final width with no delayed animation |
| Semantics | Use accessible progress semantics; label and numeric value remain visible in text |

### `AvatarGlow`

| Property | Contract |
|----------|----------|
| Core purpose | Add premium presence to avatars without obscuring the face or fallback glyph |
| Ring treatment | Thin inner edge plus soft outer emerald halo; hero/candidate cards stay readable first |
| Surface contrast | Avatar edge must stay distinct from both dark and panel backgrounds |
| Motion | Soft pulse or shimmer only if subtle; reduced motion falls back to static glow |
| Scope | Allowed on hero-facing and candidate surfaces, not on every nav or shell avatar |

### `StepProgress`

| Property | Contract |
|----------|----------|
| State system | Completed = gold, active = emerald, pending = muted |
| Interaction | Presentational only in Phase 16; users do not click nodes to jump steps |
| Density | Render compact nodes for all questions while preserving a separate `Step X of Y` text readout |
| Shape | Nodes should read as gem/hex markers, not plain circles with generic SaaS styling |
| Layout | Drop into onboarding without rewriting fetch, answer, or submit logic |

## Upgraded Consumer Contracts

### `RaidCard`

| Property | Contract |
|----------|----------|
| Hover spotlight | Implement only here using CSS custom properties `--glow-x` and `--glow-y` on the card root |
| Pointer model | Update spotlight from pointer movement without React render-state tracking |
| Fallback behavior | On touch/coarse pointers, keep the emerald hover ring and skip live spotlight tracking |
| Surface mood | Strongest Phase 16 visible lift: luminous but restrained emerald hover, layered panel feel, no neon arcade treatment |
| Composition | Use extracted `FitScoreBadge` and `HexSlotIndicator`; keep raid-specific mapping and copy inside the feature component |
| CTA | Button label becomes `Join Raid`; keep it visually primary but smaller than page-level hero CTAs |

### `HeroCard`

| Property | Contract |
|----------|----------|
| Surface emphasis | Calmer than `RaidCard`; no cursor-following spotlight |
| Avatar | Consume `AvatarGlow` as the default portrait treatment |
| Hierarchy | Hero name and headline remain the primary text read path; chips and score are secondary |
| Skill treatment | Prefer `SkillBar` or denser readable skill composition over more decorative ornaments |
| Compatibility | Keep current status badge, mini radar, and role mapping behavior intact |

### `HexRadar`

| Property | Contract |
|----------|----------|
| Animation strategy | Reveal the profile polygon with draw-on motion; do not scale or rotate the whole chart as the main effect |
| Mini variant | Stay quieter than full-size radar; preserve readability in candidate and hero summary cards |
| Reduced motion | Render final state immediately |
| Overlay handling | Target/desired profile overlay remains secondary and gold-toned |
| Existing behavior | Preserve labels, tooltips, `mini`/`full` variants, and current data contract |

### `RollHeroReveal`

| Property | Contract |
|----------|----------|
| Reveal sequencing | Preserve the current staggered Framer Motion sequence; only refine visual maturity |
| Card treatment | Elevated but calmer than `RaidCard`; no cursor spotlight |
| Avatar treatment | Use `AvatarGlow` for stronger character presence |
| Score treatment | Prefer `FitScoreBadge` and disciplined mono score readouts over oversized decorative numbers |
| Action copy | Primary action reads `Invite Hero`; secondary action remains `Re-roll` |

### `D20RollAnimation`

| Property | Contract |
|----------|----------|
| Scope | Visual-only polish in Phase 16 |
| Timing | Keep the established `~1.2s` roll duration |
| Surface treatment | Use premium gold/emerald framing and cleaner stage copy, not extra FX layers |
| Reduced motion | Immediate completion or minimal static state remains the accessibility fallback |
| Deferment | No sound, particles, or richer choreography before Phase 18 |

## Interaction Guardrails

- `RaidCard` spotlight is the only approved pointer-follow interaction in this phase.
- Do not add continuous hover tracking to `HeroCard`, `RollHeroReveal`, onboarding containers, or shell surfaces.
- New component motion is reveal-only and one-time; no new looping theatrical animations.
- Any new animated surface must respect `prefers-reduced-motion: reduce`.
- Focus styles must remain visible on buttons, card links, and tooltip triggers; decorative glow cannot replace contrast-safe focus indication.
